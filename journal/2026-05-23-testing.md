## Mocking approach

### Unit + integration

Each test creates a fresh JSDOM via `loadApp()` and overrides
`dom.window.fetch` with a `vi.fn` returning a `Promise<Response-like>`.
The mock response shape mirrors each provider's documented payload
(see `tests/fixtures/responses.js`). Request bodies are captured in
closure to assert wire-format compliance.

### E2E

Playwright's `page.route()` intercepts requests to:

- `https://api.anthropic.com/v1/messages`
- `https://api.openai.com/v1/chat/completions`
- `https://generativelanguage.googleapis.com/*` (regex)

The mock chooses what to return based on the role of the call (source,
judge, or synthesis) detected from prompt content. Helpers live in
`tests/e2e/helpers/mock-providers.js`.

## Why JSDOM + direct `jsdom` package, not Vitest's `environment: 'jsdom'`

Vitest's jsdom environment runs in the same global context as the test
module. When I appended a `<script>` element with the application's
inline code, the DOM updated but the script did not execute (verified
by `console.log` of `window.escapeHTML` returning `undefined`). I tried
toggling `environmentOptions.jsdom.runScripts = 'dangerously'`,
recreating script elements via `document.createElement('script')` (not
`innerHTML`), appending to `<head>` instead of `<body>`, and using
`window.eval(script)` — none worked.

The fix: switch to `environment: 'node'`, import `JSDOM` from the
`jsdom` package, and instantiate per-test domains. This gives the same
DOM API surface, with full control over `runScripts`, and the
application's inline script executes correctly. Trade-off: tests can't
use `globalThis.document` directly; they must go through `dom.window`.
That's a deliberate clarity win — each test owns its document and
there's no implicit shared state.
